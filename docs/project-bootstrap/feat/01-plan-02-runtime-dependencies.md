### Install runtime dependencies (REQ-004)

Run, in this order so the lockfile keeps related packages grouped:

```bash
npm install \
  zod@^4 \
  kysely@^0.28 \
  kysely-postgres-js@^3 \
  postgres@^3 \
  @supabase/supabase-js@^2 \
  @supabase/ssr@^0.6 \
  @anthropic-ai/sdk@^0.66 \
  pdf-parse@^1
```

Notes per package:
- **zod ^4** — schema validator, the base for every domain Zod schema (per [domain-model REQ-002](../../domain-model/spec/spec.md#L27)). Major-pinned per RN-004.
- **kysely ^0.28** — query builder. Per [ADR-001](../../domain-model/feat/01-plan-01-primitives.md#L47), Kysely is chosen over Prisma/Drizzle.
- **kysely-postgres-js ^3** — Kysely dialect for the `postgres` driver. Per ADR-015.
- **postgres ^3** — Postgres.js driver (peer of `kysely-postgres-js`).
- **@supabase/supabase-js ^2** — Supabase client (used by future auth flows + service-role queries).
- **@supabase/ssr ^0.6** — App Router auth helpers. Future FE specs use this.
- **@anthropic-ai/sdk ^0.66** — Anthropic SDK for the extractor in [requisitos-extraction](../../requisitos-extraction/spec/spec.md). Major version pinned per [requisitos-extraction REQ-018](../../requisitos-extraction/spec/spec.md#L57). The exact major is recorded as a `// SDK_MAJOR=<n>` comment in `lib/extraction/anthropic/config.ts` when that file is created (T2 of requisitos-extraction).
- **pdf-parse ^1** — PDF text extractor for [pdf-ingestion ADR-004](../../pdf-ingestion/spec/spec.md#L189). Locked-in choice.

### Install dev dependencies (REQ-005)

```bash
npm install --save-dev \
  vitest@^4 \
  @vitest/coverage-v8@^4 \
  @types/node@^20 \
  tsx@^4
```

Notes per package:
- **vitest ^4** — test runner.
- **@vitest/coverage-v8 ^4** — coverage provider matched to vitest's major.
- **@types/node ^20** — Node 20 typings (matches `engines.node`).
- **tsx ^4** — for running ad-hoc TypeScript scripts (occasionally useful for migration helpers; not on the critical path).

`@types/react`, `@types/react-dom`, `typescript`, `eslint`, and `eslint-config-next` are already added by `create-next-app` in T1 — verify they are present and at the major versions indicated in REQ-004/005.

```bash
npm ci  # MUST succeed; if it doesn't, the lockfile drifted from package.json
```

### Verify zero peer-dep warnings (NFR-05)

```bash
npm ls --all 2>&1 | grep -E '^(npm warn|npm err|UNMET)'
```

Expected: zero matches. If any appear, resolve before continuing (typically by bumping a peer's range or pinning a transitive resolver).

### Author ADR-014 (REQ-011, RN-009)

Write `.nybo/foundation/adrs/ADR-014-npm-package-manager.md`:

```markdown
# ADR-014: npm as the package manager

## Status
Accepted (2026-04-27)

## Context
The repo's CI workflow at `.github/workflows/ci.yml` already uses `npm ci` and
`actions/setup-node@v4` with `cache: 'npm'`. AGENTS.md documents `npm run`
scripts. Specs authored before bootstrap (domain-model NFR-01) referenced
`pnpm typecheck`, creating a single inconsistency that needed resolving.

## Decision
Use **npm** (not pnpm or yarn). Lockfile is `package-lock.json`. CI uses
`npm ci`. All scripts are `npm run <script>`. The single pnpm reference in
domain-model NFR-01 is converged on `npm run typecheck` via project-bootstrap
REQ-012 / T6.

## Alternatives Considered
- **pnpm** — faster installs, content-addressable disk usage, stricter
  hoisting. Rejected: the existing CI + AGENTS.md investment in npm is real
  (every spec authored before bootstrap referenced npm scripts implicitly via
  AGENTS.md). Switching pays the churn cost of editing CI, AGENTS.md, every
  spec, and converting the lockfile, with no offsetting product benefit at
  this stage. Reconsidered for v2 if install/CI duration becomes a felt
  constraint.
- **yarn (classic v1)** — deprecated; package authors are migrating off it.
- **yarn berry / pnp** — Plug'n'Play breaks too many tools (vitest, ESLint
  plugins). Not worth the friction.
- **Bun** — interesting future option (very fast); not yet a stable choice
  for production Next.js apps.

## Consequences
- (+) Zero churn against existing CI + AGENTS.md.
- (+) `npm` is universal — every contributor has it without separate install.
- (−) Slower installs than pnpm; larger `node_modules`. Not a measurable
  bottleneck at this size.
- (−) No workspace-level dependency dedup beyond what npm provides natively.
  Not a concern for a single-package repo.

## References
- [project-bootstrap RN-001](../../docs/project-bootstrap/spec/spec.md#L94)
- [requisitos-extraction REQ-018](../../docs/requisitos-extraction/spec/spec.md#L57) (major-version pinning convention)
```

### Design Rationale (Single Responsibility)

T2 owns one concern: the dependency graph. Every other concern (Supabase init, vitest config, CI workflow, source code) is in another task. This separation lets a future "swap dependency X for Y" change touch only T2 without disturbing the rest of the bootstrap spec or its evidence.
